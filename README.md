
# MediTrack Connect - 全院中心化数据库管理工具

本项目是一个专为医疗机构设计的 **远程数据库管理工具**。采用纯粹的远程 MySQL 架构与物理分层存储逻辑，实现全院范围内的数据同步、闭环随访及附件云端化共享。

---

## 一 系统架构与核心定位

1. **远程数据库驱动**：程序无本地数据存储。所有终端直接接入同一台远程 MySQL 服务器，实现多点并发、实时同步。
2. **中心化存储引擎**：PDF 报告及影像扫描件由全院共享的物理存储路径（支持 UNC）统一管控，由管理员通过数据库下发配置，全院透明调阅。
3. **物理分拣逻辑**：文件自动按 `[患者ID] / [报告类别] / [日期_文件名]` 逻辑进行物理分拣，确保海量病历附件的有序组织。
4. **实时分析工作台**：具备自动化数据统计引擎，直接读取远程数据库流水，实时聚合档案总量、闭环完成率及月度趋势。

---

## 二 服务器端配置与环境要求

### 1 MySQL 中心库配置
- **推荐版本**：MySQL 8.0.x
- **监听端口**：默认为 `10699`（请在防火墙中开启入站规则）。
- **Schema 同步**：程序首次连接成功后将自动维护 7 张核心业务表。

### 2 中心附件存储 (UNC 共享)
- **物理建议**：在中心存储服务器（或 NAS）创建共享文件夹。
- **配置路径**：管理员在程序设置中配置 UNC 路径（如 `\\172.17.16.18\MediStorage`）。
- **权限要求**：确保所有运行本程序的 Windows 账户对该路径具有读/写/修改权限。

---

## 三 核心业务表结构定义

系统自动维护以下核心表：

- **SP_USERS**: 人员权限表。存储操作员及 Admin 安全凭据。
- **SP_SETTINGS**: 全院全局配置。包含系统名称及物理存储路径 `STORAGE_PATH`。
- **SP_PERSON**: 患者电子病历档案主表。存储基本信息，作为业务流的唯一事实来源。
- **SP_ZYJG**: 重要异常结果流水表。记录临床发现、风险分类及预定触发日期。
- **SP_SF**: 随访闭环结案流水。存储回访结论及复查核实情况。
- **SP_LOGS**: 全量业务审计日志。记录关键操作的时间戳与经办人。
- **SP_DOCUMENTS**: 附件索引表。关联物理文件路径、上传日期与报告分类。

## 四 客户端部署与加固

### 1 运行环境
- **操作系统**：Windows 7 / 8.1 / 10 / 11。
- **关键依赖**：必须预装 **Microsoft Visual C++ Redistributable 2015-2022**。
- **兼容性模式**：程序针对 Windows 7/8.1 默认禁用硬件加速 (GPU) 及沙盒模式，解决白屏及无反应问题。

### 2 诊断与回顾
- **轨迹记录**：所有 SQL 执行流及物理文件 IO 错误均实时记录至 `%APPDATA%/meditrack-connect/app.log`。
